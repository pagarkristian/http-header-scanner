# v3.5 — Cookie Security Analysis

## Feature Name
Cookie Security Analysis

## Purpose
Check every cookie a target website sets (`Set-Cookie` headers) for the security
attributes that matter most: `Secure`, `HttpOnly`, `SameSite`, and an expiry.

- `src/cookie_analyzer.py`

## Functions Added
- `cookie_analyzer._get_raw_cookies(response)` — reads every raw `Set-Cookie` header
  without merging them (see "Common Mistakes" below for why this matters).
- `cookie_analyzer._parse_cookie(raw_cookie)` — splits one raw `Set-Cookie` string
  into name + attributes.
- `cookie_analyzer._evaluate_cookie(cookie)` — checks a parsed cookie against
  security best practices and returns a list of issues.
- `cookie_analyzer.analyze_cookies(response)` — public entry point; returns a list
  of analyzed cookies.
- `display.display_cookie_analysis(cookies)`
- `html_report._build_cookie_table(cookies)`

## New Concepts Learned

### Cybersecurity Concepts
- **Secure flag** — the cookie is only sent over HTTPS, never plain HTTP.
- **HttpOnly flag** — JavaScript (`document.cookie`) cannot read the cookie, which
  blocks a large class of XSS-based cookie theft.
- **SameSite attribute** — controls whether the cookie is sent on cross-site
  requests, which is a core defense against CSRF attacks.
- **SameSite=None requires Secure** — browsers reject `SameSite=None` cookies that
  aren't also `Secure`, so checking for that combination catches a real
  misconfiguration, not just a style preference.

### Python Concepts
- **Why not `response.headers.get("Set-Cookie")`** — `requests` collapses repeated
  headers into one comma-joined string, which breaks when a server sends more than
  one cookie, or when a cookie's `Expires` value contains a comma. Reading from
  `response.raw.headers.getlist("Set-Cookie")` avoids that.
- **Defensive parsing** — every attribute is checked with `.lower()` before
  comparison, since browsers/servers write `Secure`, `secure`, `SECURE`
  inconsistently.

## How the Feature Works Internally

### Flow Diagram
```
response.raw.headers.getlist("Set-Cookie")
        |
        v
_parse_cookie()   -->  {name, secure, httponly, samesite, has_expiry, raw}
        |
        v
_evaluate_cookie() -->  + {issues: [...], status: "Secure" | "Needs Attention"}
        |
        v
analyze_cookies()  -->  list[dict], one entry per cookie
        |
        v
main.py  -->  display_cookie_analysis() (CLI) + export_to_json() (JSON) + html_report (HTML)
```

## Common Mistakes
- **Using `response.headers.get("Set-Cookie")`.** This silently breaks multi-cookie
  responses because of the comma-joining behavior described above.
- **Case-sensitive attribute checks.** `if "Secure" in attribute` fails to catch
  `secure` written in lowercase. Always normalize with `.lower()` first.

## How I Can Explain This Feature During an Interview
"I added a cookie security checker that reads the raw Set-Cookie headers instead of
the merged ones requests normally gives you, because merging breaks parsing when a
site sets multiple cookies. For each cookie I check Secure, HttpOnly, SameSite, and
expiry, and flag specific issues like SameSite=None without Secure, which is an
actual browser-rejected misconfiguration, not just a nice-to-have."

## Questions an Interviewer Might Ask

**Q: Why is HttpOnly important if Secure is already set?**
A: They protect against different threats. Secure stops the cookie from being sent
over unencrypted HTTP (network-level threat). HttpOnly stops JavaScript from reading
the cookie at all, which protects against XSS-based cookie theft even on a page
served over HTTPS.

**Q: What happens if a cookie has SameSite=None but not Secure?**
A: Modern browsers reject the cookie entirely — it won't even be set. My analyzer
flags this as a specific issue because it's not just a best-practice suggestion,
it's a configuration that will actually break in the browser.
