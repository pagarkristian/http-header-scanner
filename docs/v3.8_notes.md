# v3.8 — Release Candidate

## Feature Name
Release Candidate Review

## Purpose
Do a final consistency pass across the whole project before calling it "Version 3":
fix version-number mismatches, fix a missing dependency, tidy `.gitignore`, and fill
in packaging metadata.

## Files Added
None.

## Files Modified
- `requirements.txt` — **bug fix:** `requests` was missing entirely (only `rich` was
  listed), even though `scanner.py` imports it. A fresh `pip install -r
  requirements.txt` would have failed to run the scanner at all.
- `src/main.py` — `--version` now prints `v3.8` (was hardcoded `v3.0`); fixed a typo
  ("Profesional" → "Professional") in the CLI description.
- `.gitignore` — added project-specific runtime output (`config.ini`, `logs/*.log`,
  `reports/*/`) so generated files don't get committed, while keeping `.gitkeep`
  placeholders tracked.
- `pyproject.toml` — was empty; now has project metadata (name, version,
  dependencies, entry point) so the project can be installed with `pip install .`.

## What Was Reviewed (and found clean)
- **Imports** — every module imports only what it uses; no circular imports (data
  flows one direction: `constants` → `analyzer`/`cookie_analyzer` → `scoring` →
  `display`/`exporter`/`html_report`).
- **Naming** — consistent `snake_case` for functions/variables, `SCREAMING_CASE` for
  constants, across all modules.
- **Folder organization** — matches what's documented in `README.md` and
  `docs/architecture.md`.
- **Tests** — `tests/*.py` are still empty placeholders. Left as-is rather than
  rushing shallow tests in for the sake of it; listed properly in
  `docs/roadmap.md` as a real next step instead.

## New Concepts Learned

### Cybersecurity Concepts
- **Supply chain hygiene.** A missing dependency in `requirements.txt` is a small
  thing, but it's the same category of issue as an unpinned or malicious dependency
  — your declared dependencies should be the *complete, accurate* list of what the
  code actually needs. Reviewing `requirements.txt` against actual `import`
  statements is a quick, real check worth doing before any release.

### Python Concepts
- **`pyproject.toml`** — the modern standard way to declare a Python project's
  metadata and dependencies (replacing the older `setup.py` approach). Filling it
  in enables `pip install .` and `project.scripts` (a console entry point).
- **Auditing for consistency programmatically.** Instead of manually re-reading
  every file, a couple of targeted `grep` searches (e.g. for old version strings)
  caught the leftover `v3.0` and `1.10` mismatches quickly and reliably.

## How the Review Was Done (Flow)

```
1. grep for known-stale strings (old version numbers, typos)
        |
        v
2. py_compile every .py file (catches syntax errors)
        |
        v
3. Re-run the full export_to_json + generate_html_report pipeline end-to-end
        |
        v
4. Diff requirements.txt against actual `import` statements in src/
        |
        v
5. Clean up generated artifacts (__pycache__, config.ini, reports/*) before packaging
```

## Common Mistakes
- **Bumping the version string in one place and forgetting the others.** This
  project had the version in three places (`--version` output, JSON report
  `"version"` field, and now `pyproject.toml`) that had drifted out of sync. Worth
  grepping for the old value whenever bumping a version, rather than trusting memory.
- **Forgetting to check `requirements.txt` against real imports.** It's easy to add
  a new `import` during development and forget to also add it to
  `requirements.txt`, since your own machine already has it installed from earlier
  testing.

## How I Can Explain This Feature During an Interview
"For the release candidate pass, I didn't just re-read the code — I grepped for
known-stale strings like old version numbers, ran `py_compile` on every file to
catch syntax issues, and cross-checked `requirements.txt` against the actual
`import` statements in the codebase, which caught a real bug: `requests` was
missing from the dependency list entirely, so a clean install would have been
broken."

## Questions an Interviewer Might Ask

**Q: How did you find the missing `requests` dependency?**
A: I compared every `import` statement across `src/` against what
`requirements.txt` declared. `scanner.py` imports `requests`, but it wasn't listed
— easy to miss by just reading the file once, but a systematic check catches it.

**Q: Why keep `tests/` empty instead of writing quick tests to fill them?**
A: Rushed tests that don't meaningfully verify behavior are worse than no tests,
because they create false confidence. I'd rather document it honestly as a known
gap in the roadmap and write real tests for `analyzer.py`, `scoring.py`, and
`cookie_analyzer.py` properly, since those three have no external dependencies and
are the easiest to test well.
