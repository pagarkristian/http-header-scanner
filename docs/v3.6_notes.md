# v3.6 — Application Improvements

## Feature Name
Logging, Configuration File, Better Error Handling

## Purpose
Make the app easier to operate and debug: settings live in a config file instead of
hardcoded values, every scan is recorded to a log file, and unexpected errors show a
clean message instead of crashing with a raw traceback.

## Files Added
- `src/config.py`
- `config.ini` (auto-generated on first run; not committed with real values, but a
  default one is created automatically so the app works out of the box)

## Files Modified
- `src/logger_config.py` — was an empty placeholder; now implements `setup_logger()`
  and `get_logger()`.
- `src/scanner.py` — `scan_website()` now accepts a `timeout` argument, auto-adds
  `https://` if the user forgets the scheme, and logs failures.
- `src/utils.py` — `create_report_directory()` now accepts a `reports_dir` argument
  instead of hardcoding `"reports"`.
- `src/display.py` — new `display_unexpected_error()` function.
- `src/main.py` — loads config at startup, sets up the logger, passes `config` and
  `logger` into `scan_target()`, and wraps the whole scan flow in a `try/except`.

## Functions Added
- `config.load_config()` — reads `config.ini`, creating it with defaults first if
  missing.
- `config._create_default_config()` — internal helper that writes the default file.
- `logger_config.setup_logger(logs_dir, log_level)` — configures the shared logger.
- `logger_config.get_logger()` — retrieves the already-configured logger from other
  modules (used in `scanner.py`).
- `display.display_unexpected_error(url, error)`

## Functions Modified
- `scanner.scan_website(url)` → `scan_website(url, timeout=10)`
- `utils.create_report_directory(url)` → `create_report_directory(url, reports_dir="reports")`
- `main.scan_target(url)` → `scan_target(url, config, logger)`

## New Concepts Learned

### Cybersecurity Concepts
- **Fail closed, log everything.** A scanning tool that crashes on one bad target in
  a 500-target batch file is a liability. Catching unexpected errors and logging them
  (instead of stopping the whole run) is a basic operational-security habit.
- **Least-surprise defaults.** Auto-adding `https://` when a user forgets the scheme
  matches what real scanners (and browsers) do, and avoids accidentally scanning
  plain HTTP by default.

### Python Concepts
- **`configparser` (standard library, no extra install needed).** Settings are
  grouped into `[sections]` with `key = value` pairs, read back with
  `config["section"]["key"]` or typed getters like `.getint()`.
- **`logging.FileHandler` + guarding against duplicate handlers.** Calling
  `setup_logger()` once per target (in a batch scan) would attach a new handler each
  time without the `if logger.handlers: return logger` guard, causing every log line
  to be written multiple times.
- **Broad `except Exception` as a last resort.** Normally you catch specific
  exceptions. Here, at the very top of the scan flow, a broad catch is intentional —
  it's the last line of defense so one bug doesn't take down an entire batch scan.
  The real detail is preserved via `logger.exception()`, which logs the full
  traceback even though the user only sees a short message.

## How the Feature Works Internally

### Flow Diagram
```
main() start
   |
   v
load_config()  -->  reads config.ini (creates it with defaults if missing)
   |
   v
setup_logger(logs_dir, log_level)  -->  logs/scanner.log ready
   |
   v
scan_target(url, config, logger)
   |
   +-- try:
   |     scan_website(url, timeout=config timeout)
   |     ... normal flow ...
   |     logger.info("Finished scan...")
   |
   +-- except Exception:
         logger.exception(...)          # full traceback -> log file
         display_unexpected_error(...)  # short message -> user
```

## Common Mistakes
- **Forgetting the duplicate-handler guard in `setup_logger()`.** In a batch scan
  (`--file targets.txt`), `scan_target()` runs once per line — but `setup_logger()`
  is only called once in `main()`, precisely to avoid this problem.
- **Catching `Exception` too early.** If the broad `try/except` had wrapped only
  `scan_website()` instead of the whole function, bugs in `analyze_headers()` or
  `export_to_json()` would still crash the batch. The catch needs to wrap the entire
  per-target flow to be a real safety net.

## How I Can Explain This Feature During an Interview
"I moved hardcoded values like timeout and output folder into a config.ini file
using Python's built-in configparser, added file-based logging so failed scans are
recorded even in a large batch run, and wrapped the scan flow in a broad exception
handler so one bad target can't crash the rest of the batch — the full error still
gets logged, but the user sees a clean message instead of a traceback."

## Questions an Interviewer Might Ask

**Q: Why use `configparser` instead of a `.env` file or `pyyaml`?**
A: `configparser` is in the Python standard library, so it needs no extra
dependency and no internet access to install anything. For a project this size, a
`.ini` file with a few settings doesn't need YAML's extra features.

**Q: Isn't `except Exception:` considered bad practice?**
A: Generally yes, because it can hide bugs. I only use it at the outermost level of
the per-target scan, as a last-resort safety net for a batch tool, and I always log
the full exception with `logger.exception()` so nothing is silently lost — the user
just doesn't see a raw traceback on screen.

**Q: What's the difference between the CLI error panels and the log file?**
A: The panels (`display_scan_error`, `display_unexpected_error`) are short,
user-facing messages meant to be read on screen right now. The log file
(`logs/scanner.log`) is a permanent, detailed record — including full tracebacks —
meant for debugging after the fact.
