# v3.7 — Project Polish

## Feature Name
Project Polish

## Purpose
Turn a working tool into a presentable portfolio project: a README someone can
actually follow, a real changelog, an architecture doc, and working example files.

## Files Added
- `docs/architecture.md`
- `docs/roadmap.md`
- `docs/screenshots/README.md` (placeholder + instructions — see note below)

## Files Modified
- `README.md` — rewritten with features, installation, usage, configuration,
  project structure, header/cookie reference tables.
- `CHANGELOG.md` — filled with the full v3.0 → v3.8 version history.
- `examples/single_target.txt` — was empty; now has one example URL.
- `examples/sample_report.json` — was empty; regenerated using the real
  `export_to_json()` pipeline so it reflects the actual v3.6 JSON schema
  (headers + cookies + summary), not hand-written fake data.

## New Concepts Learned

### Cybersecurity Concepts
- No new security concepts this milestone — this is a documentation-focused
  milestone.

### Python Concepts
- **Generating realistic example data from real code, not by hand.**
  `examples/sample_report.json` was produced by actually calling
  `analyze_headers()`, `analyze_cookies()`, and `export_to_json()` against a fake
  response object, instead of hand-typing JSON. This guarantees the example always
  matches the real output schema — if the schema changes later, the old example
  would look obviously stale instead of silently lying.

## Honest Note on Screenshots
This milestone's environment has no internet access, so a real scan (and a real
terminal screenshot) couldn't be captured here. Instead of faking one,
`docs/screenshots/README.md` explains exactly how to generate real screenshots
yourself in a few steps. Faking a "screenshot" would be misleading in a portfolio
project — real evidence matters more than a placeholder image.

## Common Mistakes
- **Writing example/sample data by hand.** It drifts out of sync with the real
  code the moment a field is renamed. Generating it from the real export function
  (as done here) avoids that entirely.
- **Copy-pasting version numbers into multiple places.** Before this milestone,
  `main.py --version` said "v3.0" while the JSON report said `"version": "1.10"` —
  neither matched the actual project version. Fixed as part of v3.8's consistency
  pass.

## How I Can Explain This Feature During an Interview
"For the documentation milestone, I generated the example JSON report by actually
running the export function against test data instead of hand-writing JSON, so the
example can't silently drift out of sync with the real schema. I also wrote an
architecture doc that explains the data flow between modules, which is something I
found genuinely useful for onboarding myself back into the project after a break."

## Questions an Interviewer Might Ask

**Q: Why not just fake a screenshot for the README?**
A: A portfolio project should represent real, verifiable work. A faked screenshot
would misrepresent what the tool actually looks like running, so I documented exact
steps to generate a real one instead.

**Q: How do you keep documentation from going stale as the code changes?**
A: Where possible, generate examples from the real code path instead of writing
them by hand — like `sample_report.json` here. For things that can't be
auto-generated (like the README), I try to review them again whenever the relevant
code changes, which is exactly what the v3.8 milestone did.
