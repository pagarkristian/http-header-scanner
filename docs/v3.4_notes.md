# v3.4 — Advanced Security Headers

## Feature Name
Advanced Security Headers (Cross-Origin Isolation headers + legacy cross-domain header)

## Purpose
Expand the set of HTTP security headers the scanner checks from 6 to 10, adding modern
cross-origin isolation headers that real-world scanners (Mozilla Observatory,
SecurityHeaders.com) also check.

## Files Added
- `docs/v3.4_notes.md` (this file)

## Files Modified
- `src/constants.py`
  - Added 4 headers to `SECURITY_HEADERS`
  - Added `HEADER_INFORMATION` entries for the 4 new headers
  - Rebalanced `HEADER_WEIGHTS` so the total still sums to 100

## Functions Added
None. This milestone only added data (constants), not new functions.

## Functions Modified
None. `analyze_headers()`, `calculate_score()`, and all `display.py` functions are
generic — they loop over `SECURITY_HEADERS` / `HEADER_WEIGHTS` instead of hardcoding
header names, so they picked up the new headers automatically with zero code changes.

## New Headers Added

| Header | What it does |
|---|---|
| `Cross-Origin-Opener-Policy` (COOP) | Stops other origins from keeping a JavaScript reference to this page's window. |
| `Cross-Origin-Embedder-Policy` (COEP) | Requires cross-origin resources to explicitly opt in before this page can load them. |
| `Cross-Origin-Resource-Policy` (CORP) | Controls whether other origins are allowed to load this site's resources (images, scripts). |
| `X-Permitted-Cross-Domain-Policies` | Restricts legacy plugins (Flash, Acrobat) from making cross-domain requests. |

## New Concepts Learned

### Cybersecurity Concepts
- **Cross-Origin Isolation** — COOP and COEP work together to isolate a page from other
  origins at the browser level, mitigating side-channel attacks like Spectre.
- **Direction of protection matters** — `X-Frame-Options` protects against this page being
  *embedded* by others (clickjacking). `COOP` protects against other windows *holding a
  reference* to this page (the opposite direction).
- **Legacy attack surface still matters** — `X-Permitted-Cross-Domain-Policies` targets old
  plugin technology (Flash/Acrobat), but is still checked because legacy systems are still
  running in the wild.

### Python Concepts
- **Data-driven design** — header names, risk descriptions, and scoring weights all live in
  one dictionary-based config file (`constants.py`), completely separate from the logic that
  uses them (`analyzer.py`, `scoring.py`). This is why adding 4 new headers required editing
  only one file.
- **DRY (Don't Repeat Yourself)** — because every module reads from `SECURITY_HEADERS` /
  `HEADER_WEIGHTS` instead of hardcoding header names, there was only one place to update.

## How the Feature Works Internally

### Before
```
SECURITY_HEADERS = [ 6 headers ]
HEADER_WEIGHTS totals 100 across 6 headers
```

### After
```
SECURITY_HEADERS = [ 10 headers ]
HEADER_WEIGHTS totals 100 across 10 headers (existing weights reduced proportionally)
```

### Flow Diagram
```
constants.py (SECURITY_HEADERS, HEADER_WEIGHTS, HEADER_INFORMATION)
        |
        v
analyzer.py  -->  loops over SECURITY_HEADERS, checks each against response.headers
        |
        v
scoring.py   -->  loops over analysis, sums HEADER_WEIGHTS for headers that are Present
        |
        v
display.py / exporter.py / html_report.py  -->  render whatever analysis contains
```
Because every downstream step loops instead of hardcoding, adding a header to
`SECURITY_HEADERS` (plus its weight and info) is enough to make it show up everywhere:
CLI table, HTML report, JSON report, and the score calculation.

## Common Mistakes
- **Forgetting to rebalance weights.** If you only add new weights without reducing the old
  ones, the total exceeds 100 and the score calculation breaks (scores above 100).
- **Adding a header to `SECURITY_HEADERS` but forgetting `HEADER_WEIGHTS` or
  `HEADER_INFORMATION`.** This causes a `KeyError` in `scoring.py` or `main.py` at runtime.
  Always add a header to all three places together.

## Side Effect to Know About
A site that previously had all 6 old headers and scored 100 will now score around 81
(Medium risk) unless it also has the 4 new headers. This isn't a bug — the scanner is just
holding sites to a more complete, modern standard now.

"I extended the scanner to check 4 additional modern security headers related to
cross-origin isolation — COOP, COEP, CORP, and X-Permitted-Cross-Domain-Policies. Because
the project was already designed with the header list and scoring weights as data in a
config file, rather than hardcoded in the logic, adding new headers only required editing
one file. I also had to rebalance the scoring weights to make sure they still summed to 100,
so the score stays on a consistent 0–100 scale."

## Questions an Interviewer Might Ask

**Q: Why does `Cross-Origin-Opener-Policy` matter if `X-Frame-Options` already exists?**
A: They protect in opposite directions. `X-Frame-Options` stops your page from being framed
by someone else. `COOP` stops other windows from holding a reference to your page's window
object, which prevents cross-window attacks like Spectre-based side channels.

**Q: How did you decide the weight for each new header?**
A: I judged each header's security impact relative to the existing ones — COOP/COEP protect
against browser-level side-channel attacks, so they got a moderate weight, while the legacy
`X-Permitted-Cross-Domain-Policies` header protects against a much narrower, older attack
surface, so it got the smallest weight (2). Then I adjusted all weights so the total stayed
at 100.

**Q: What would break if you added a new header to `SECURITY_HEADERS` but forgot to add it
to `HEADER_WEIGHTS`?**
A: `scoring.py` would throw a `KeyError` because it looks up `HEADER_WEIGHTS[header]` for
every header in the analysis result. That's why all three constants (headers list, weights,
info) need to stay in sync.
